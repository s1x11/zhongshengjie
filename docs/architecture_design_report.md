## 执行摘要

本报告为《众生界》AI小说创作辅助系统提供整体架构设计与长期发展规划。系统采用**模块化单体架构**向**微服务就绪架构**演进的设计思路，通过清晰的限界上下文划分、统一的数据流设计和可扩展的插件机制，支持从当前单机创作工具向多用户协作平台的平滑演进。

**核心设计决策**:
- 架构模式: 模块化单体（当前）→ 微服务（未来）
- 数据存储: Qdrant向量数据库 + 文件系统混合存储
- 通信模式: 同步API调用为主，事件驱动为辅
- 扩展策略: 水平扩展向量检索层，垂直扩展业务逻辑层

## 1. 现状分析

### 1.1 系统现状

《众生界》是一个基于多Agent协作的小说创作辅助系统，核心特性包括：

| 维度 | 现状 |
|------|------|
| **架构模式** | 模块化单体，分层架构 |
| **核心团队** | 5位作家Agent + 1位评估Agent |
| **数据规模** | 38万+案例、986条技法、160条设定 |
| **技术栈** | Python + Qdrant + BGE-M3 + Claude Skills |
| **部署方式** | 本地单机运行 |
| **用户规模** | 单用户创作 |

### 1.2 现有架构分层

```
┌─────────────────────────────────────────────────────────────┐
│                    对话入口层 (Conversation)                   │
│   意图识别 → 意图澄清 → 状态检查 → 数据新鲜度检测              │
├─────────────────────────────────────────────────────────────┤
│                    核心工作流 (8阶段)                         │
│   需求澄清→大纲解析→场景识别→经验检索→设定检索→场景契约→创作→评估 │
├─────────────────────────────────────────────────────────────┤
│                    支撑系统层                                 │
│   变更检测 │ 类型发现 │ 统一检索 │ 反馈系统 │ 生命周期管理      │
├─────────────────────────────────────────────────────────────┤
│                    数据层                                     │
│   Qdrant向量库 │ 配置文件 │ 章节日志 │ 评估报告               │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 当前问题识别

| 问题类别 | 具体问题 | 影响 | 优先级 |
|----------|----------|------|--------|
| **耦合度** | core/目录下模块间循环依赖风险 | 维护困难 | P1 |
| **扩展性** | 新增作家Agent需修改多处代码 | 扩展成本高 | P1 |
| **数据流** | 数据回流闭环不完整 | 无法持续学习 | P0 |
| **状态管理** | 工作流状态分散存储 | 难以恢复 | P1 |
| **配置管理** | 配置分散在5+个文件 | 易出错 | P2 |
| **测试覆盖** | 部分模块缺乏单元测试 | 质量风险 | P2 |

## 2. 目标架构设计

### 2.1 架构愿景

设计一个能够支持以下演进的架构：

1. **短期（3-6个月）**: 完善模块化，消除循环依赖，建立清晰的模块边界
2. **中期（6-12个月）**: 支持多项目/多世界观并行创作
3. **长期（1-2年）**: 可选支持多用户协作、云端部署


### 3.2 数据存储策略

| 数据类型 | 存储方式 | 理由 |
|----------|----------|------|
| 向量数据 | Qdrant | 高效相似度检索 |
| 配置数据 | JSON文件 | 易于版本控制 |
| 创作内容 | Markdown文件 | 人工可读可编辑 |
| 日志数据 | JSON文件 | 结构化查询 |
| 状态数据 | SQLite/JSON | 轻量级持久化 |
| 缓存数据 | 内存/Redis(未来) | 高性能访问 |

## 7. 架构决策记录

### ADR-001: 采用模块化单体架构

**状态**: 已接受

**上下文**:
当前系统是一个单机创作工具，团队规模小（1-2人），需要快速迭代。微服务架构会带来不必要的运维复杂度。

**决策**:
采用模块化单体架构，通过清晰的模块边界和接口设计，为未来可能的微服务拆分预留空间。

**后果**:
- 更容易: 开发、测试、部署
- 更难: 独立扩展特定模块

### ADR-002: 使用Qdrant作为向量数据库

**状态**: 已接受

**上下文**:
需要支持百万级文本片段的相似度检索，要求毫秒级响应。

**决策**:
使用Qdrant作为向量数据库，支持Dense+Sparse+ColBERT混合检索。

**后果**:
- 更容易: 高性能向量检索、本地部署
- 更难: 需要维护额外的服务依赖

### ADR-003: 数据回流闭环设计

**状态**: 提议

**上下文**:
系统需要从评估结果和用户反馈中持续学习，但目前缺乏完整的数据回流机制。

**决策**:
建立三级数据回流：评估反馈→技法库更新、用户修改→经验沉淀、技法效果→推荐优化。

**后果**:
- 更容易: 系统持续改进、个性化推荐
- 更难: 需要设计阈值机制和冲突解决策略

### ADR-004: 插件化扩展机制

**状态**: 提议

**上下文**:
需要支持新增作家Agent、自定义评估维度等扩展需求。

**决策**:
定义WriterPlugin、RetrievalPlugin、EvaluationPlugin标准接口，支持动态加载。

**后果**:
- 更容易: 功能扩展、第三方集成
- 更难: 接口设计需要足够抽象
